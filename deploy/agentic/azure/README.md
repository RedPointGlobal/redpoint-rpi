# RPI v7.7 — Agentic Deployment on Azure

Automated end-to-end deployment of Redpoint Interaction (RPI) on Azure. The RPI Deploy Agent provisions infrastructure, deploys the Helm chart, initializes the cluster, and hands off a running environment -- no manual steps required.

## How It Works

The deploy agent is available in two forms:

- **Claude Code**: `/deploy-rpi` slash command -- runs in your terminal
- **Helm Assistant**: agent endpoint -- runs via the web UI or API

Tell the agent what you need in plain language:

```
/deploy-rpi Deploy RPI on Azure in East US 2, SQL Server, 
Realtime API with MongoDB cache and Service Bus queues, 
private ingress on redpointcdp.com
```

The agent handles everything else.

## What the Agent Does

### Phase 0: Infrastructure
Provisions the full Azure landing zone using Bicep (`deploy/agentic/azure/main.bicep`):

| Resource | Name Pattern | Purpose |
|:---------|:-------------|:--------|
| Resource Group | `rg-rpi-{6char}` | Contains all resources |
| AKS Automatic | `aks-rpi-{6char}` | Kubernetes with Workload Identity, CSI Secret Store |
| Managed Identity | `id-rpi-{6char}` | Pod authentication to Key Vault |
| Azure SQL Server | `sql-rpi-{6char}` | Operational + Logging databases |
| Key Vault | `kv-rpi-{6char}` | All app secrets (SDK mode) |
| Service Bus | `sb-rpi-{6char}` | Standard tier with 6 RPI queues |
| Storage Account | `strpi{6char}` | Azure Files for file output |
| App Gateway for Containers | `agc-rpi-{6char}` | Ingress with TLS termination |
| Private Endpoints | 4 PEs | SQL, Key Vault, Service Bus, Storage |
| VNet (if new) | `vnet-rpi-{6char}` | AKS, AGC, and PE subnets |

### Phase 1: Discovery
The agent asks you:
- Namespace name
- Container registry and image tag
- Domain name (public or private)
- Is Realtime API needed? If yes: which cache provider? which queue provider?

### Phase 2: Pre-Checks
The agent verifies:
- Cluster connectivity and namespace
- Image pull secret exists (you pre-create this)
- Workload Identity federation is configured
- Key Vault contains all required secret keys (names only, never reads values)
- AGC and ALB controller are running
- CSI Secret Store driver is active

### Phase 3: Generate
The agent produces a Helm overrides file. It never generates secrets -- all app credentials live in Key Vault.

### Phase 4: Deploy
The agent runs `helm install` and monitors pod startup. If pods fail, it pulls logs, diagnoses, and fixes before proceeding.

### Phase 5: Validate
The agent calls `POST /api/deployment/installcluster` to create databases, schema, and the initial admin user. It then verifies health endpoints on all services.

### Phase 6: Handoff
The agent reports:
- Initial admin credentials (username `coreuser`, auto-generated password)
- All service URLs
- What was deployed and configured
- Next steps (download RPI Client, apply license, configure data warehouse connections)

## Security Model

| Concern | How it's handled |
|:--------|:-----------------|
| App secrets (DB credentials, connection strings, API tokens) | Key Vault + Workload Identity (SDK mode). Agent never sees values. |
| Image pull secret | You pre-create it. Agent verifies it exists. |
| TLS certificate | Key Vault -> CSI Secret Store sync -> K8s Secret `ingress-tls` via validation pod |
| Internal passwords (Redis, RabbitMQ) | Auto-generated by the Helm chart |
| Ingress | Application Gateway for Containers (no NGINX) |

The agent never asks for, handles, or logs credentials. It verifies Key Vault key names exist without reading their values.

## Prerequisites

Before running the agent, ensure you have:

1. **Azure subscription** with Contributor access
2. **Sufficient VM quota** for AKS Automatic (min 16 vCPUs of D-series in target region)
3. **Container images** mirrored to your registry (ACR or other)
4. **Image pull secret** created in the target namespace
5. **TLS certificate** uploaded to Key Vault (the agent syncs it to the cluster via CSI)

If deploying into an existing VNet, you also need:
- AKS subnet (min /16)
- AGC subnet (min /24, delegated to `Microsoft.ServiceNetworking/trafficControllers`)
- PE subnet (min /24)
- Private DNS zones for `privatelink.database.windows.net`, `privatelink.vaultcore.azure.net`, `privatelink.servicebus.windows.net`, `privatelink.file.core.windows.net`

## Key Vault Secrets Checklist

The agent verifies these keys exist in Key Vault (populated by the Bicep template for database secrets, or by you for other services):

**Auto-populated by Bicep (infrastructure provisioning):**
- `ConnectionString-Operations-Database`
- `ConnectionString-Logging-Database`
- `Operations-Database-Server-Password`
- `Operations-Database-ServerHost`
- `Operations-Database-Server-Username`
- `Operations-Database-Pulse-Database-Name`
- `Operations-Database-Pulse-Logging-Database-Name`
- `SMTP-Password` (empty)
- `RealtimeAPI-ServiceBus-ConnectionString` (if Service Bus created by Bicep)
- `RealtimeAPI-MongoCache-ConnectionString` (if internal MongoDB deployed by chart)

**Auto-populated by agent (Phase 5):**
- `RPI-Admin-Username` -- initial admin user (`coreuser`)
- `RPI-Admin-Password` -- auto-generated, change after first login
- `RealtimeAPI-Auth-Token` -- auto-generated UUID
- `RPI-Tenant-ClientID` -- first tenant's client ID
- `RPI-Tenant-Name` -- first tenant's name
- `RPI-License-ActivationKey` -- placeholder until user provides the actual key

**You provide after deployment:**
- `RPI-License-ActivationKey` -- update with your actual license activation key, then run `/deploy-rpi resume`

## Resume Mode

After deployment, the Execution Service and Node Manager will be unhealthy until the license is activated. The agent defers license activation because the key is typically not available at deployment time.

Once you have the license key:

```bash
# 1. Store the key in Key Vault
az keyvault secret set --vault-name <kv-name> --name RPI-License-ActivationKey --value <your-key>

# 2. Run the agent in resume mode
/deploy-rpi resume
```

The agent reads the key from Key Vault, activates the license, and verifies all services become healthy.

Additional resume commands:
- `/deploy-rpi resume` -- activate license
- `/deploy-rpi resume add-tenant <name>` -- add another tenant
- `/deploy-rpi resume status` -- check deployment health

```bash
az group delete --name rg-rpi-<6char> --yes --no-wait
```
